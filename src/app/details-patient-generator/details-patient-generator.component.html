Patient Record Generator

One of the challenges in healthcare IT User Interface design is knowing what the designs will look and behave like when populated with real data. The type and amount of data can greatly affect how usable and readable the designs are.

This is the motivation behind the patient data generator. I created this tool for designers creating static designs, and also for folks creating coded prototypes. For static prototypes, you can download an SVG file of the patient record. For coded prototypes, you can download a json file containing the patient data. This tool uses techniques similar to a Monte Carlo simulation to create a realistic patient record. The generator is architected to be infinitely extensible through a set of data/configuration files (mostly csv files) that can all be edited to tweak the generator results.

Details; how the generator works

The generation process starts with a problem profile, and the various parts of the record are created based on the profile. For example, using a diabetes profile, there are a certain number of problems defined, namely: diabetes mellitus, neuropathy, foot ulcer, elevated PSA, Upper Respiratory Infection, vitamin b, and iron. Obviously vitamin b and iron are considered more medications and not problems, so they are ignored for problems and only used when picked a medication. As mentioned above, this profile definition can be edited and tweaked.

For each problem in the profile, a specific ICD-10 item is selected from a file of all ICD-10 items that could be associate with the general problem profile. For example the profile problem 'diabetes mellitus' has about 150 specific ICD-10 problems to choose from. This design means that each diabetes patient record will contain slightly different diabetes (and other) problems.

Also, for each profile problem, there is a medication file containing the appropriate medications prescribed for that problem. As with the problems, there are many similar medications for each problem, so that each generated patient record would contain different medications for each problem.

This approach is true for all of the remaining parts of the record as well. A set of vital signs is defined for each profile as well, based on the types of vitals usually collected for that type of patient. Appropriate vital sign value ranges are defined, and a random value is generated within that range. Specifically, the configuration file for the vital sign defines a mean and standard deviation, and then random, normally distributed values are chosen based on that. The mean value, for example, for a hypertension profile would be higher for blood pressure than for other profiles. Choosing random values in this way lends realism (because the values are a bit messy as in real life), and credibility (the values are believable for the patient profile). The number of sets of vitals and the date intervals can also be controlled (e.g., if you want to include a set of vitals every month for the past two years.

Patient demographic information is selected randomly from a set of about 15,000 fake patients.

All of the above factors combine to ensure that many patient records can be generated for the same profile, and they will all be a bit different.

Finally, regarding performance, a single patient record can typically be generated in less than half a second. This is of course system dependent, but the time would be approximately that.
