Clinical Risk Items Web Application

This is an example of a project in which I performed these roles:

  • Requirements gathering
  • User research and exploration
  • Iterations of prototyped designs using both Sketch and real code, with user and stakeholder feedback
  • Front-end UI development in Angular
  • Back-end development using Swift running on Ubuntu

Previous Process

The previous process for patient safety issues used a complex and confusing Excel spreadsheet duplicated across product areas.

New Web-based Application

I proposed a centralized web application that I would write in Angular to make the process much simpler for users to enter information, and also simpler for product administrators to get aggregated reports.

The web application solves all of the usability problems of the spreadsheet method. By hosting the application on an internal web server, all patient safety issues are now consolidated into one place.

Any patient safety issue can quickly be located using the global search, or using the navigation organized by product area.

I worked with stakeholders to rank order the information for a given safety item, and use that to determine what fields would be surfaced in the top level list.

Entering a new patient safety issue is very easy and provides all necessary guidance to the user when and where they need it. And the user can progressively enter the risk and remediation at a future time, as users do not always have that information initially. The design keeps the UI clean by not surfacing those data fields unless the user indicates they want to enter them.

Here the user has indicated they want to enter initial risk values, and so that default page is shown here, with the user selecting a value for 'initial likelihood'

For beginning users, they can check a checkbox to show help text for all of the fields. This clarifies the intent of all of the fields for new users, albeit at the expense of a bit noisier UI.
